Enhancement 3: Databases and Full-Stack Development

Artifact Description

▼

For Enhancement 3, I completed the transformation of the Hospital Management System from a console-based application into a production-ready full-stack web application. This final enhancement integrated MongoDB for persistent data storage, built a Spring Boot REST API backend, and created a React dashboard for the front-end interface. The enhancement also included implementing JWT authentication for secure user access.

The original artifact stored all data in memory using Java arrays, meaning all information was lost when the program closed. This limitation made the system unsuitable for real-world use. Enhancement 3 addressed this fundamental flaw by implementing a complete database layer with MongoDB, transforming the application into a genuine enterprise-ready system.

Justification for Inclusion

▼

I selected this artifact for Enhancement 3 because database integration is essential for any production application, and healthcare systems particularly demand reliable data persistence, scalability, and security. This enhancement allowed me to demonstrate comprehensive full-stack development skills and database management expertise.

Database Design and Implementation

I designed and implemented a MongoDB database schema with four collections:

  • Doctors Collection: Stores physician information including specialty, office visit fees, and patient visit counts
  • Patients Collection: Maintains patient records with demographic information and primary care physician assignments
  • Appointments Collection: Tracks scheduling data with references to doctors and patients
  • Employees Collection: Manages hospital staff information, roles, and payroll data

Why MongoDB?

I chose MongoDB (NoSQL) over a traditional relational database because:

  • Flexible Schema: Healthcare data models evolve frequently. MongoDB's document-based structure allows adding fields without schema migrations
  • Horizontal Scalability: MongoDB scales out easily as patient volumes grow
  • JSON-like Documents: Natural fit for REST APIs and modern JavaScript front-ends
  • Rapid Development: Faster iteration during development without complex schema changes

RESTful API Development

I built a comprehensive REST API using Spring Boot that provides:

  • Complete CRUD operations for all entities
  • Proper HTTP status codes and error handling
  • CORS configuration for secure cross-origin requests
  • Input validation and data sanitization
  • Structured JSON responses for front-end consumption

Front-End Development

The React dashboard demonstrates modern front-end capabilities:

  • Real-time data visualization using Recharts library
  • Interactive forms for managing hospital records
  • Responsive design for desktop and mobile devices
  • Professional UI following Material Design principles

Security Implementation

I implemented JWT (JSON Web Token) authentication providing:

  • Secure user login with token generation
  • Password encryption using BCrypt
  • Protected API endpoints requiring valid authentication
  • Role-based access control foundation

The enhancement improved the artifact by making it a complete, deployable application suitable for real hospital use. The system now persists data reliably, serves multiple concurrent users through a web interface, and maintains security best practices.

Reflection on the Enhancement Process

▼

Learning and Growth

Enhancement 3 represented the most significant learning curve of all three milestones. Integrating a database, building a REST API, and creating a React front-end required synthesizing knowledge from multiple domains. Prior to this enhancement, I had worked with each technology individually, but never integrated them into a complete full-stack application.

The most valuable lesson was understanding how data flows through a modern web application. Starting from a user clicking a button in React, triggering an API call to Spring Boot, which queries MongoDB and returns formatted data, then updating the UI with the response. This end-to-end understanding of the request-response cycle is fundamental to full-stack development.

I also gained deep appreciation for the importance of consistent data models across layers. The structure of data in MongoDB must align with the entity classes in Java, which must match the expected format in React components. Mismatches at any layer cause integration failures. Maintaining this consistency required careful planning and documentation.

Challenges Encountered

Challenge 1: MongoDB Connection Management

Initially, I struggled with MongoDB connection pooling and resource management. My early implementation opened a new connection for every database operation, which was inefficient and caused connection exhaustion under load.

Solution: I implemented proper connection pooling using Spring Data MongoDB's built-in MongoClient configuration. The configuration establishes a connection pool at application startup, reuses connections across requests, properly handles connection timeouts and retries, and implements graceful shutdown to prevent data loss.

Challenge 2: Cross-Origin Resource Sharing (CORS)

When connecting the React front-end (running on localhost:3000) to the Spring Boot API (localhost:8080), I encountered CORS errors preventing the browser from making requests.

Solution: I configured CORS in Spring Boot by creating a global CORS configuration that allows requests from the React development server, specifies allowed HTTP methods (GET, POST, PUT, DELETE), enables credentials for authentication, and sets appropriate headers for API responses.

Challenge 3: Asynchronous Data Loading in React

Managing asynchronous API calls in React components was initially confusing. State updates triggered re-renders at unexpected times, causing infinite loops or stale data displays.

Solution: I implemented proper React patterns by using useEffect hooks with dependency arrays to control when API calls execute, implementing loading states to provide user feedback during data fetches, properly handling errors with try-catch blocks and user-friendly error messages, and using async/await for cleaner asynchronous code.

Challenge 4: Data Validation Across Layers

Ensuring data integrity required validation at multiple points: client-side in React forms, server-side in Spring Boot controllers, and at the database level with MongoDB schemas.

Solution: I implemented a three-tier validation strategy:

  1. Client-Side: React form validation for immediate user feedback
  2. Server-Side: Spring Boot validation annotations ensuring API integrity
  3. Database-Side: MongoDB schema validation as the final safeguard

This defense-in-depth approach prevents invalid data from entering the system at any level.

Challenge 5: JWT Authentication Integration

Implementing JWT authentication required understanding token generation, validation, and refresh mechanisms. I also needed to integrate authentication across both the API and React front-end.

Solution: I created a comprehensive authentication system with a login endpoint that generates JWT tokens, a token validation filter that protects API endpoints, token storage in localStorage on the client side, and automatic token inclusion in API requests via Axios interceptors.

Incorporation of Feedback

Throughout development, I received feedback on code organization and error handling:

  1. Separating Concerns: Initially, I had database logic mixed with controller logic. I refactored to use service classes that handle business logic, keeping controllers thin and focused on HTTP concerns.
  2. Error Handling Improvement: Early API implementations returned generic error messages. I enhanced error handling to return specific, actionable error information including HTTP status codes that match the error type, detailed error messages for debugging, and sanitized error responses that don't expose system internals.
  3. API Response Standardization: Feedback indicated inconsistent response formats made front-end integration difficult. I standardized all API responses to follow a consistent structure with data payload, status information, and error details when applicable.
  4. Code Documentation: I added comprehensive documentation including API endpoint documentation with examples, database schema documentation, setup instructions for running the full stack, and inline comments explaining complex logic.

Database Schema and Design

▼

Doctors Collection

{
  _id: ObjectId,
  name: String,
  specialty: String,
  address: String,
  officeVisitFee: Number,
  patientVisits: Number
}

Patients Collection

{
  _id: ObjectId,
  name: String,
  address: String,
  primaryCarePhysician: String,
  medicalRecordNumber: String
}

Appointments Collection

{
  _id: ObjectId,
  doctorId: String,
  patientId: String,
  appointmentDate: Date,
  appointmentTime: String,
  status: String,
  priority: String
}

Employees Collection

{
  _id: ObjectId,
  name: String,
  role: String,
  email: String,
  hourlyRate: Number,
  monthlyHours: Number
}

Indexing Strategy

  • Doctors: Index on specialty for filtered searches
  • Patients: Index on medicalRecordNumber for quick lookup
  • Appointments: Compound index on doctorId and appointmentDate for conflict detection
  • Employees: Index on email for authentication queries

REST API Endpoints

▼

Doctor Endpoints

  • GET /api/doctors - Retrieve all doctors
  • GET /api/doctors/{id} - Get specific doctor
  • POST /api/doctors - Create new doctor
  • PUT /api/doctors/{id} - Update doctor information
  • DELETE /api/doctors/{id} - Remove doctor

Patient Endpoints

  • GET /api/patients - Retrieve all patients
  • GET /api/patients/{id} - Get specific patient
  • POST /api/patients - Register new patient
  • PUT /api/patients/{id} - Update patient information
  • DELETE /api/patients/{id} - Remove patient record

Appointment Endpoints

  • GET /api/appointments - Retrieve all appointments
  • GET /api/appointments/doctor/{doctorId} - Get doctor's appointments
  • POST /api/appointments - Schedule new appointment
  • PUT /api/appointments/{id} - Update appointment
  • DELETE /api/appointments/{id} - Cancel appointment

Employee Endpoints

  • GET /api/employees - Retrieve all employees
  • POST /api/employees - Add new employee
  • PUT /api/employees/{id} - Update employee information

Reports and Authentication

  • GET /api/reports/payroll - Generate payroll report
  • POST /api/auth/login - User authentication
  • POST /api/auth/register - User registration

System Architecture

▼

Three-Tier Architecture

Presentation Layer (React)

  • Runs on http://localhost:3000 (development)
  • Handles user interface and user interactions
  • Makes HTTP requests to API layer

Application Layer (Spring Boot)

  • Runs on http://localhost:8080
  • Processes business logic
  • Validates and sanitizes data
  • Manages authentication and authorization

Data Layer (MongoDB)

  • Runs on default port 27017
  • Persists all application data
  • Supports aggregation queries and indexing

Data Flow

  1. User interacts with React UI
  2. React sends HTTP request to Spring Boot API
  3. Spring Boot validates request and checks authentication
  4. Service layer processes business logic
  5. Repository layer queries MongoDB
  6. Data flows back through layers to React
  7. React updates UI with response data
